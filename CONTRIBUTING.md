# Contributing to PerpetualSoftware/unraid-templates

Notes for **maintainers** adding or updating templates in this repo. End-user install instructions live in [`README.md`](README.md).

## How to add a new template

1. Create `<app>/` directory at repo root.
2. Drop in:
   - `<app>/<app>.xml` — the Unraid CA Container template (Unraid Docker XML, schema `Container version="2"`)
   - `<app>/icon.png` — 256x256 PNG, LANCZOS-downsampled from the app's source-of-truth icon
   - `<app>/README.md` — app-specific notes (form fields, env vars, backup pattern, etc.)
3. Update [`README.md`](README.md)'s "Apps" table with a row for the new app.
4. Set both URLs in the new template:
   - `<TemplateURL>https://raw.githubusercontent.com/PerpetualSoftware/unraid-templates/main/<app>/<app>.xml</TemplateURL>`
   - `<Icon>https://raw.githubusercontent.com/PerpetualSoftware/unraid-templates/main/<app>/icon.png</Icon>`
5. Run the format conformity checks below.
6. Run [Submission to Community Applications](#submission-to-community-applications) Validate + Scan.
7. Open a PR. After merge, the portal picks up the new template on its next refresh.

## Icon source-of-truth

There are two distinct icons in this repo:

- **Vendor icon** at `/icon.png` — referenced by `ca_profile.xml`'s `<Icon>`. This is the **Perpetual Software company mark**, fetched from `https://perpetualsoftware.org/icon_transparent.png` (the favicon/header logo on perpetualsoftware.org). When the brand changes there, regenerate this file.
- **Per-app icon** at `<app>/icon.png` — referenced by `<app>/<app>.xml`'s `<Icon>`. This is the **product mark** (e.g. Pad's icon for `pad/icon.png`). The source-of-truth lives in the product's own repo (e.g. [`PerpetualSoftware/pad`](https://github.com/PerpetualSoftware/pad)'s `web/static/icon-512.png` for Pad).

### Square-source pipeline (per-app icons typically)

```bash
python3 -c "from PIL import Image; \
  Image.open('<source>.png').resize((256, 256), Image.LANCZOS).save('<app>/icon.png', 'PNG', optimize=True)"
```

### Non-square source pipeline (vendor icon, sometimes per-app)

If the source isn't square, scale-to-fit and pad with transparency so CA's listing tile renders correctly:

```python
from PIL import Image
src = Image.open('<source>.png').convert('RGBA')
target = 256
scale = target / max(src.size)
resized = src.resize((int(src.width * scale), int(src.height * scale)), Image.LANCZOS)
canvas = Image.new('RGBA', (target, target), (0, 0, 0, 0))
canvas.paste(resized, ((target - resized.width) // 2, (target - resized.height) // 2), resized)
canvas.save('icon.png', 'PNG', optimize=True)
```

The vendor icon was generated this way from the 549x367 source.

## dockerMan SAVE format conformity

The Unraid wiki ([Docker Template Schema](https://wiki.unraid.net/DockerTemplateSchema)) is explicit:

> "the only supported XML format is that which is generated by Unraid's docker tab when hitting SAVE on the template. Any manual editing of the XML files may present compatibility issues for CA and if they turn out to be incompatible, will result in the template being blacklisted within CA until brought into conformity with the dockerMan generated files."

Keep template XML aligned with what dockerMan emits when an operator hits **Apply** on the install form. If you're editing this file by hand, validate by:

1. Sideload your edited copy via the Route A path documented in [`README.md`](README.md).
2. Open it in CA, hit **Apply** to install it.
3. Diff your edited copy against `/boot/config/plugins/dockerMan/templates-user/<container-name>.xml` (the post-SAVE serialized version).
4. Migrate any structural diff (added/removed/renamed elements, attribute-form changes) back into the canonical file. Cosmetic diffs (line endings, em dash → numeric entity) are XML-equivalent and OK to leave.

### Things to know about dockerMan's serializer

These bit us on the first round of Pad's template — capture once, pay forward:

- **XML comments don't survive a SAVE round-trip** — dockerMan strips `<!-- ... -->`. Don't embed maintainer notes inside template XML; put them in the per-app README instead.
- **`<Description>` is not a recognized field** — dockerMan drops it. `<Overview>` is the canonical CA-displayed text.
- **BBCode (`[b]...[/b]`, `[i]...[/i]`, etc.) is stripped from `<Overview>`** — use plain text for emphasis.
- **Empty `<Config>` elements should be self-closing** (`<Config .../>` not `<Config></Config>`).
- **The schema includes empty marker elements** — `<MyMAC/>`, `<ReadMe/>`, `<Requires/>`, `<TailscaleStateDir/>` etc. dockerMan emits them all even when empty.
- **Em dashes** should be encoded as `&#x2014;` numeric entities (not literal Unicode `—`) — that's what dockerMan SAVE outputs.

Last verified against Unraid 7.x dockerMan output on 2026-05-06.

## ca_profile.xml schema crib

The repo-level [`ca_profile.xml`](ca_profile.xml) declares vendor-wide metadata. The portal docs at <https://ca.unraid.net/submit/help/repository-info-xml> *imply* a `<CommunityApplications>` root, but **real-world submitters use `<Maintainer>` as the root element**. Verified against:

- [`JakeShirley/unraid-templates`](https://github.com/JakeShirley/unraid-templates)
- [`Mavrag/unraid-templates`](https://github.com/Mavrag/unraid-templates)
- [`skrashevich/unraid-templates`](https://github.com/skrashevich/unraid-templates)
- [`slamanna212/UnraidTemplates`](https://github.com/slamanna212/UnraidTemplates)

Schema (fields used in this repo plus optional ones for reference):

```xml
<?xml version="1.0" encoding="utf-8"?>
<Maintainer>
  <Icon>https://raw.githubusercontent.com/PerpetualSoftware/unraid-templates/main/icon.png</Icon>
  <Profile>
    <!-- REQUIRED, non-empty. Multi-line OK. Describes what this repo holds. -->
  </Profile>
  <WebPage>https://getpad.dev</WebPage>
  <!-- Optional fields documented but not currently used: -->
  <!-- <Forum>https://forums.unraid.net/topic/<thread>/</Forum> -->
  <!-- <DonateLink>...</DonateLink> -->
  <!-- <DonateText>...</DonateText> -->
  <!-- <Discord>...</Discord> -->
  <!-- <Reddit>...</Reddit> -->
  <!-- <Twitter>...</Twitter> -->
  <!-- <Photo>...</Photo> -->
  <!-- <Video>...</Video> -->
</Maintainer>
```

The portal blocks submission if `ca_profile.xml` is missing or `<Profile>` is empty. The other fields are recommended but optional.

## Submission to Community Applications

The canonical 2026-vintage submission process is the **portal at <https://ca.unraid.net/submit>**. Older paths (PR to Squid's appfeed, Settings → CA Plugin in-Unraid form) are deprecated.

### Initial repo submission

1. Sign in to <https://ca.unraid.net/submit> (Unraid forum auth).
2. Click **Add Repository** — paste `https://github.com/PerpetualSoftware/unraid-templates`.
3. Click **Validate**. Read any errors / warnings carefully.
4. Fix flagged issues in this repo.
5. Click **Scan** (more thorough — checks image pullability, URL resolution, etc.).
6. Fix flagged issues. Loop until both Validate AND Scan return zero errors.
7. Click **Submit**. Click **Confirm**.
8. Wait for CA review (community lore: 2–7 days typical).

Save the portal's response. If anything new shows up (e.g., the portal demands a field we haven't documented), update this CONTRIBUTING.md so future-us doesn't trip on the same wire.

### Adding a template to an already-approved repo

After the initial submission lands and the repo is registered with the portal:

1. Add the new template via the [How to add a new template](#how-to-add-a-new-template) flow.
2. Open a PR; merge.
3. Re-run **Validate** + **Scan** in the portal — they read from `main`, so the new template is picked up automatically.
4. No re-submission needed. The CA appfeed refreshes the listing on its next pull (typically within a day).

### Common rejection reasons (defended against)

- **Repo private/archived** → repo is public + active.
- **Missing license** → Apache 2.0 file at root.
- **Empty `<Profile>` in `ca_profile.xml`** → check before merging.
- **Image not publicly pullable** → `docker pull <image>` from any unauthenticated host.
- **Broken icon URL** → keep all icons under `raw.githubusercontent.com/PerpetualSoftware/unraid-templates/...`.
- **Malformed XML** → caught by Validate; also catchable locally via `python3 -c "import xml.etree.ElementTree as ET; ET.parse('<file>.xml')"`.

## Verification protocol

Local-side check that catches drift the portal doesn't necessarily flag:

1. Sideload your edited copy on a real Unraid box (Route A or B per [`README.md`](README.md)).
2. Click **Apply** in the install form.
3. SSH to the box and read `/boot/config/plugins/dockerMan/templates-user/<container-name>.xml` — that's the post-SAVE serialized version dockerMan wrote.
4. Diff it against your edited copy.
5. Migrate any structural diff back into the source-of-truth template. Cosmetic-only diffs are OK to leave.

This catches things like accidental BBCode in `<Overview>`, comments that survived your editor but won't survive dockerMan, mistyped attribute forms — all the dockerMan SAVE behaviors documented above.
